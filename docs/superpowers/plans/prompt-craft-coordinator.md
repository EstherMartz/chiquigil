# Claude Code prompt — Discord Craft Coordinator

Run Claude Code from the `ffxiv-helper` repo root and paste the prompt below.

---

Implement a new **`/craft` crafting-coordinator feature** for the Discord bot in `bot/`. The full design is in `docs/superpowers/plans/2026-05-25-discord-craft-coordinator.md` — read it first and follow its File Map and Tasks. Build it task-by-task, write Vitest tests as you go, and run `npm run typecheck` + `npm test` from `bot/` before declaring done.

**Goal:** A guild member runs `/craft new item:"Grade 8 Tincture of Strength" qty:8` and the bot replies with an announcement embed that breaks the request down Teamcraft-style — what to **craft** (grouped by crafter job), what to **buy** off the market board, what to buy from **gil vendors**, what to buy with a **currency** (which tomestone/scrip + cost), and what to **gather** — then lets people **claim** line items and **log progress** with buttons/menus, editing the embed live. Projects and assignments persist across bot restarts.

**Reuse existing code — do not reinvent it:**
- Recursion source: `bot/src/loadSnapshots.ts` exposes `snapshots.recipes: Map<itemResultId, Recipe>` where `Recipe.ingredients = {itemId, amount}[]` and `Recipe.classJob` is a crafter code (`CRP`/`BSM`/… or `'ANY'`). Walk this map to explode a target to leaf materials.
- Source classification is already written: `src/features/shoppingList/shoppingListSurvey.ts` → `surveyIngredients(demand, prices, vendorMap, shopSnapshot)` returns per-item `{ mb, npc, currency, autoSource }`. The `currency` field IS the "buy with which currency" answer. Reuse it.
- Currency defs: `src/lib/currencies.ts`. Gil vendors: `snapshots.vendorMap`. Market fetch: `bot/src/fetchMarketForOutputs.ts` (use the `dc`/Chaos `MarketData`; the cached layer in `bot/src/chat/tools.ts` avoids hammering Universalis). Name search/autocomplete: `bot/src/chat/nameIndex.ts` (`buildNameIndex`, `searchItems`).

**First, extend snapshot loading** (`bot/src/loadSnapshots.ts`): also load `public/data/snapshots/specialShop.json` (`{ bakedAt, byCurrency: [CurrencyId, ShopEntry[]][] }` → `SpecialShopSnapshot`) and `gathering.json` (`{ bakedAt, entries: [itemId, GatheringInfo][] }` → `Map<number, GatheringInfo>`). Mirror the baked shapes in `src/lib/staticSnapshots.ts` (`loadStaticSpecialShopSnapshot`, `loadStaticGatheringCatalog`) and the types in `src/lib/specialShopSnapshot.ts` + `src/lib/gatheringCatalog.ts`. Add `specialShop` and `gatheringCatalog` to `BotSnapshots`.

**Build `bot/src/craft/`:**
- `explode.ts` — pure recursive breakdown: target → `{ crafts: Map<id,{outputQty,craftCount,job}>, leaves: Map<id,qty> }`. Depth cap (~20) + on-path cycle guard. **Handle recipe yield:** `const yield = recipe.amountResult ?? 1; const craftCount = Math.ceil(qtyNeeded / yield);` and scale each sub-ingredient's demand by `craftCount` (NOT by output units) — otherwise raw-mat demand is ~3× too high for the many intermediates that yield 3. The shared `Recipe` type already carries `amountResult?` (the `AmountResult` fetch field + parser were added to `src/lib/recipes.ts` and `src/lib/recipeSnapshot.ts`, with tests — do NOT redo that). Add a `craftIntermediates` flag (default true); when false, intermediates become leaves to buy. Unit-test it.
- `sourcing.ts` — fetch market prices for leaf ids, call `surveyIngredients`, and tag each leaf with one source: `gather` (in the gathering catalog, no trivially cheap vendor), else `currency`, else `vendor` (when `autoSource==='npc'`), else `market`. Crafts become `source:'craft'` tasks with `meta.job`. Produce a `Breakdown = { crafts, acquire }` of `CraftTask`s. Unit-test the priority rules.
- `store.ts` — **better-sqlite3** store at `config.craftDbPath` (`process.env.CRAFT_DB_PATH ?? 'data/craft.db'`). Tables `projects` and `tasks` exactly as in the plan's SQL. Functions: `openCraftStore`, `createProject`, `addTasks`, `getProject`, `getTasks`, `listOpenProjects`, `claimTask`, `logProgress` (adds to `qty_done`, auto-sets `done` at 100%), `unclaimTask`, `setProjectMessageId`, `closeProject`. Wrap claim/progress in a transaction. Test against a `:memory:` db.
- `render.ts` — `buildProjectMessage(project, tasks)` → `{ embeds, components }`. Group line items under CRAFT (by job) / BUY — Market Board / BUY — Currency / GATHER, each showing assignee + `done/needed`. Respect the 25-option select cap and embed field limits (paginate/collapse if needed).
- `commands.ts` — handlers for `/craft new|list|show|close|setup` (with `item` autocomplete via `searchItems`). `setup` (admin, `ManageMessages`) posts + pins the "Request a craft" prompt and initialises the roll-up board in the craft channel. `new` resolves the item, runs the breakdown, persists the project+tasks, posts the embed, stores its message id, and optionally pings a role. `close` is restricted to the creator or `ManageMessages` (mirror the `/purge` check in `index.ts`).
- `interactions.ts` — handle component customIds of the form `cproj:<projectId>:<action>[:<taskId>]` (`claim` via a string select, `progress` via a modal, `unclaim`, `refresh`) and edit the announcement message in place.

**Dedicated crafting channel (#crafting) + extras:**
- Config (`bot/src/config.ts`): add `craftChannelId = optional('CRAFT_CHANNEL_ID')` and `crafterRoleId = optional('CRAFT_ROLE_ID')` (same pattern as the existing `chatChannelId`). `/craft` stays runnable from any channel, but the announcement board always posts to the craft channel (fall back to the invoking channel if unset).
- Pinned roll-up board: maintain one bot-owned pinned "Active crafting projects" message listing every open project + progress %, re-rendered/edited on every project create/claim/progress/close. Add `buildBoardMessage(openProjects)` to `render.ts` and a `channel_state` table (`guild_id, channel_id, board_message_id, request_message_id`) to `store.ts`.
- Thread per project: after posting an announcement, `await message.startThread({ name: project.name, autoArchiveDuration: 1440 })`; add a `thread_id` column to `projects`; drop claim/progress notes in the thread.
- `@Crafters` role ping: if `crafterRoleId` (or a `/craft new ping_role` option) is set, include `<@&roleId>` in the announcement `content` with `allowedMentions: { roles: [roleId] }` (never let it escalate to @everyone).
- "Request a craft" button: add a `/craft setup` admin subcommand that posts + pins a standing message with a button `cproj:request`; clicking opens a modal (item, qty, optional name) whose submit id `cproj:requestmodal` runs the same `new` flow. Add `buildRequestPrompt()` to `render.ts`.
- On `ClientReady`, if `CRAFT_CHANNEL_ID` is set, ensure the board (and request prompt, if previously set up) still exist via `channel_state` and recreate if deleted. Note: slash commands show in every channel — don't hard-gate, just post the board to the craft channel.

**Wire it up:**
- `bot/src/registerCommands.ts`: add the `/craft` `SlashCommandBuilder` with the four subcommands and autocomplete on `item`.
- `bot/src/index.ts`: at startup, open the store and assemble craft deps `{ store, snapshots, nameIndex, cfg, fetchMarket }`. In `InteractionCreate`, route `commandName === 'craft'` to `commands.ts`, and route component interactions whose `customId.startsWith('cproj:')` to `interactions.ts` **before** the existing `decodeCustomId`/cleanup branch so the two schemes don't collide. Also handle the `cproj:requestmodal` modal submit, and on `ClientReady` ensure the pinned board/request messages exist when `CRAFT_CHANNEL_ID` is set.
- `bot/package.json`: add `better-sqlite3` + `@types/better-sqlite3`. `bot/.gitignore`: add `data/`.

**Constraints:** ESM + discord.js v14 conventions already used in the repo. Keep the breakdown engine and store pure/Discord-free so they're unit-testable.
