# Quest Items — Crafter Class Quest MB-Camping Table — Design

**Status:** Spec (pre-plan). Phase 1 of a 4-phase quest-items roadmap (crafter class quests now; custom deliveries / beast tribes / general quest items later).

## Goal

Surface every HQ/NQ turn-in item required by crafter class quests (the lv 5/10/15… milestone progression quests for the 8 crafters), sorted by Marketboard demand signals, so the user can identify "buy-for-laziness" niches — items that players reliably pay above-market for because they're stuck on a quest and want to skip the gathering/crafting step.

## Non-goals

- No custom deliveries, beast tribe quests, or general story/side quest items. Each gets its own spec; the snapshot infra built here is reused.
- No "I'm leveling Carpenter, what do I need?" planning view organized by-crafter. The MB-camping use case won (user chose flat sortable table). A per-crafter planning view can be layered on later.
- No HQ-yield prediction for the items the user might craft to satisfy a quest. The table is a market-watch view, not a craft planner.
- No automatic flagging of stale market data beyond what `pickHighestTrustedTier` already does.
- No DC/region-wide market scanning. Home world only, like Vendor Flip and Currency Flip.

## User flow

1. User clicks "Quest items" in the nav. Lands on `/quest-items`.
2. Top of the page:
   - 8 crafter-job filter chips (CRP / BSM / ARM / GSM / LTW / WVR / ALC / CUL), multi-select, all enabled by default.
   - Search box (filters by item name substring).
   - Min-listings input (numeric, default 0 — show everything; raising it hides thin markets).
   - HQ/NQ/Either toggle (default: HQ — quests usually want HQ turn-ins).
3. Below: flat sortable table of rows, one per (quest × required item). Default sort: `totalRevenue DESC`.
4. Click an item name → routes to existing `/item/:id` page (full detail, recipe modal, etc.).
5. Click a sort header → resort. Click again → reverse direction.
6. Filter chip toggle → re-renders without re-fetching the snapshot (filter is client-side over the cached snapshot + market data).

## Architecture

This follows the existing flip-tool conventions (Material Flip, Currency Flip, Vendor Flip): pure compute over a snapshot + Universalis market data, rendered through a React Query–wrapped runner.

### Data layer

- `src/lib/questSnapshot.ts` — fetches XIVAPI v2 `Quest` sheet rows filtered to crafter class quests, extracts required items. Mirrors `src/lib/leveSnapshot.ts` shape: a parser + typed `QuestSnapshot` export. Exposes `fetchQuestSnapshot()` and `parseQuestPage()` (the latter for testability over fixture pages).
- `src/features/queries/useQuestSnapshot.ts` — React Query hook with `staleTime: Infinity` (matches `useVendorShopSnapshot`, `useSpecialShopSnapshot`). Backs onto IDB cache via `recipeCache.ts`.
- `src/lib/recipeCache.ts` — bump `DB_VERSION` 9 → 10. Add `quest` object store + a `QUEST_SNAPSHOT_TS_KEY` meta key. Upgrade callback wipes the old store on v9→v10 transition so users get a fresh hydrate.
- `scripts/bake-snapshots.ts` — extend to fetch & bake `quests.json` alongside `items.json`, `recipes.json`, `gathering.json`, `leves.json`, `specialShop.json`, `vendorShop.json`. Adds ~one bake step, runs at the same time the other snapshots refresh.
- `public/data/snapshots/manifest.json` — regenerated by the bake script to include `quests` entry.

### Snapshot shape

```typescript
export interface QuestEntry {
  questId: number;
  questName: string;          // "Way of the Carpenter", "Tools of the Trade", etc.
  classJobId: number;         // 9..16 (CRP=9, BSM=10, ARM=11, GSM=12, LTW=13, WVR=14, ALC=15, CUL=16)
  level: number;              // 5, 10, 15, ..., up to current max
  requiredItems: QuestRequiredItem[];
}

export interface QuestRequiredItem {
  itemId: number;
  qty: number;
  isHqRequired: boolean;      // true if the quest requires the item to be HQ
}

export interface QuestSnapshot {
  bakedAt: number;
  entries: QuestEntry[];
}
```

### Filter for crafter class quests

XIVAPI v2 `Quest` sheet has thousands of rows. The parser filters to quests whose `ClassJobCategory` resolves to (or contains) one of the 8 crafter `ClassJob` IDs (9–16). The exact field path and link semantics depend on the actual v2 shape — see Known Risk #1 below; the implementer verifies on a fixture before writing the production filter.

The bake script logs the final per-class quest count so a regression (e.g., a future XIVAPI shape change drops all quests for one crafter) is visible in the bake output.

### Compute (pure)

```typescript
// src/features/queries/runQuestItemFlip.ts

interface QuestItemFilter {
  classJobIds: number[];       // empty = all 8 crafters
  hq: 'hq' | 'nq' | 'either';
  minListings: number;
  search: string;
}

interface QuestItemRow {
  questId: number;
  questName: string;
  classJobId: number;
  classJobAbbr: string;        // "CRP", "BSM", ... "CUL"
  level: number;
  itemId: number;
  itemName: string;
  qty: number;
  isHqRequired: boolean;
  nqPrice: number | null;      // home-world NQ trusted-tier median; null if no trusted tier
  hqPrice: number | null;      // home-world HQ trusted-tier median
  listingCount: number;        // total listings home-world (NQ + HQ)
  velocity: number;            // sales per day from Universalis
  totalRevenue: number;        // qty × (HQ price if isHqRequired else best of NQ/HQ); ranking signal
}

function runQuestItemFlip(
  snapshot: QuestSnapshot,
  itemsById: Map<number, SnapshotItem>,
  market: MarketBundle,
  filter: QuestItemFilter,
): QuestItemRow[];
```

`runQuestItemFlip` joins quest entries × required items × market data using `pickHighestTrustedTier` from `src/lib/priceTrust.ts` for both NQ and HQ tier prices. Filters before sorting. Default sort: `totalRevenue DESC, velocity DESC`.

### `totalRevenue` formula

```
priceForRanking =
  if (isHqRequired):  hqPrice ?? 0
  else:               max(hqPrice ?? 0, nqPrice ?? 0)

totalRevenue = qty × priceForRanking
```

When both `nqPrice` and `hqPrice` are null, `totalRevenue = 0` and the row is still shown (with em-dashes in the price columns) — visibility is more important than ranking accuracy for items the user might still want to scout.

### UI

- `src/features/insights/QuestItemFlipView.tsx` — page-level, owns local filter state via `useState`, calls `useQuestSnapshot`, `useItemSnapshot`, `useMarketData` (for the union of all itemIds in the snapshot — same multi-id batching `MaterialFlipView` uses), then `runQuestItemFlip`.
- `src/features/queries/QuestItemFlipResults.tsx` — the table itself. 9 columns: Lv / Job / Item / Qty / Req / NQ MB / HQ MB / Listings / Vel/day / Revenue. Item name is a `Link to="/item/:id"`. HQ glyph rendered next to qty when `isHqRequired`.
- `src/routes/QuestItems.tsx` — thin route wrapper.
- `src/App.tsx` — register `/quest-items` route.
- `src/components/Header.tsx` — add a nav link, slotted alphabetically between existing entries (implementer chooses placement).

Sort affordance reuses the existing `SortableHeader` component (already used in Material Flip and Vendor Flip).

Filter chips reuse the same Tailwind chip pattern from `LevePlanner.tsx`. Crafter abbreviation labels: `CRP / BSM / ARM / GSM / LTW / WVR / ALC / CUL` in that fixed order.

### URL params

`?jobs=9,10,11&hq=hq&min=2&q=lumber` — encodes the four filter fields so the user can share a filtered view. Matches the `queryUrlParams.ts` pattern shipped with the watchlist URL sharing.

- Top-nav link in `src/components/Header.tsx`. Label: `Quest items`.

## Testing strategy

Per repo convention, pure functions get unit tests with hand-written fixtures and view components get RTL tests with mocked hooks.

- `src/lib/questSnapshot.test.ts` (~5 tests) — parser handles single-item quests, multi-item quests, missing `isHqRequired` field (defaults to false), unknown ClassJobCategory rows filtered out, ambiguous-match union logic.
- `src/features/queries/runQuestItemFlip.test.ts` (~8 tests) — sort by totalRevenue, sort by velocity, sort by level, filter by classJobIds (empty = all), filter by hq mode (hq/nq/either), filter by minListings, filter by search substring (case-insensitive), revenue tie-break by velocity.
- `src/features/queries/QuestItemFlipResults.test.tsx` (~5 tests) — renders rows, HQ glyph appears when `isHqRequired`, em-dash on null price, sort header click works, empty state when no rows pass filter.
- `src/routes/QuestItems.test.tsx` (~3 tests) — route smoke test: mounts with mocked hooks, filter chip toggle re-renders, search input filters rows.

Target ~20 new tests; total suite stays under a minute.

## What's deferred (future phases)

- **Phase 2 — Custom deliveries.** `SatisfactionNpc` sheet; weekly rotation tracking; 14 NPCs × ~6 items each. Builds on `questSnapshot` infra by adding a parallel `deliverySnapshot.ts`.
- **Phase 3 — Beast tribe quests.** `Quest` sheet filtered by `BeastTribe` link; 14+ tribes × ~30 quests each. Reuses the `Quest` sheet parser by parametrizing the filter.
- **Phase 4 — General quest items.** All other `Quest` rows with item turn-ins. Broadest dragnet; requires filtering out unobtainable / quest-locked items so the user only sees market-buyable rows.
- **Per-crafter leveling planning view.** "I'm leveling BSM, here's my shopping list" UI organized by-crafter. Same data, different presentation.
- **Recipe expansion in-place.** Hover/click a row to show the item's recipe + ingredient costs, for self-craft vs market-buy comparison. Existing `RecipeModal` on the item detail page covers this for now.

## Known risks for implementation

1. **XIVAPI v2 `Quest` sheet shape unverified.** Per the [[reference-xivapi-v2-quirks]] memory: array-field syntax, 20k row budget, silent shape-change failures. The implementer should fetch a small sample first (~10 crafter quests) and verify field names + array decoding *before* writing the full parser. If the actual shape deviates significantly from this spec, report DONE_WITH_CONCERNS rather than improvising.
2. **`isHqRequired` flag detection.** May live in a field like `ItemRequiredHQ` or be encoded in script logic the API doesn't expose. If missing for some quests, default `isHqRequired = false` and the UI shows both NQ + HQ price columns regardless — the user can still infer from market data.
3. **Sub-quest chains with multi-item turn-ins.** Some crafter milestones have multiple required items per quest row. Stored as `requiredItems: QuestRequiredItem[]` (array). Each item becomes its own table row (joined back to its quest). One quest → N rows when N items.
4. **Universalis fetch volume.** ~128 unique items across all 8 crafters. Within one `useMarketData` call. No special handling needed.

## Open questions

None blocking. Decisions locked in via the brainstorm:

- Scope: Phase 1 = crafter class quests only.
- Use case: MB camping (flat sortable table), not crafter-leveling assistant.
- Data layer: parallel snapshot pipeline (`questSnapshot.ts` + IDB store) matching existing flip-tool patterns.
- Filter shape: `classJobIds` + `hq` + `minListings` + `search`. Encoded in URL via `?jobs=...&hq=...&min=...&q=...`.
- Default sort: `totalRevenue DESC, velocity DESC`.
- DB version bump: v9 → v10. Old IDB store wiped on first load.

If the implementer hits anything in this spec that doesn't survive contact with the real XIVAPI v2 Quest sheet, they report DONE_WITH_CONCERNS instead of improvising — the brainstorm can re-converge.
